📐 Methodology

🎯 What We Measure

We benchmark the textDocument/definition LSP request (Go to Definition) across different type checker language servers. This is one of the most commonly used IDE features for code navigation.

🔄 Test Process

  1. Clone the top Python packages from GitHub
  2. Pre-pick all random Python files and identifier positions for the run
  3. For each type checker, start a single LSP server per repository
  4. Send all "Go to Definition" requests to the same server instance
  5. Measure latency and verify if returned locations are valid

⏱️ Timeout: Requests have a 2-second timeout. Timeouts are counted as failures and excluded from latency statistics.

🔁 Server Lifecycle: One LSP server is started per type checker per repository. All test cases for that repo are sent to the same server, reflecting real-world IDE usage where a single server handles multiple requests. All type checkers receive identical test cases for fair comparison.

🔥 Warmup: An optional warmup period can be configured (via --warmup) to give the server time to index before requests are sent. Testing shows pyrefly does not benefit from warmup, so it is disabled by default.

📊 Metrics Explained

  • Latency: Time (ms) from request to response (excludes timeouts)
  • P50/P95: 50th/95th percentile latencies
  • OK Rate: Percentage of requests that completed without timeout or error (indicates reliability)
  • Success Rate: Percentage of requests that returned a valid definition location pointing to a real file (indicates accuracy)
